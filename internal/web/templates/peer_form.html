A route learned from an eBGP peer carries that peer's address as its next hop. Readvertise it over iBGP unchanged and the router at the other end has nowhere to send the traffic, unless your IGP carries the peering subnets. Leave next-hop-self on unless you know it does.
BGP never readvertises an iBGP route to another iBGP peer, which is why a plain iBGP mesh has to be full. Marking a peer a reflector client lifts that rule for them. Set a cluster ID under Settings if you run more than one reflector.
BGP makes a peer prepend its own AS number before it announces a route, so every path this peer sends should start with AS{{if .Peer.RemoteASN}}{{.Peer.RemoteASN}}{{else}}…{{end}}. If it does not, they are misconfigured or leaking someone else's routes. Turn this off for an IXP route server: a route server forwards routes without prepending itself, so leaving this on would reject everything it sends.
The second switch checks the origin AS instead: it accepts a prefix only if this peer announced it as their own. Use it when you sell someone transit but do not carry their downstreams. Their own prepending still works. To carry their downstreams too, leave it off and point an import policy at an AS set holding their expanded AS-SET.
RFC 9234 pushes leak prevention into the protocol. birdy derives the role it sends from the one above — we are a customer of an upstream, a provider to a customer, a lateral peer at an IX — and BIRD then tags routes with the Only-To-Customer attribute and rejects any that arrive as a leak. Turning it on can briefly reset the session if the far end has a conflicting role configured.
